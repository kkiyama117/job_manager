# `jm new` — flow boilerplate generator — design

**Date:** 2026-05-16
**Status:** Draft (awaiting user review)
**Reference:** `src/bin/jm.rs` (既存 CLI), `examples/simple/inputs/` (テンプレート参照元), `docs/superpowers/specs/2026-05-15-common-env-defaulting-design.md` (`.jm/` レイアウト・common 注入の前提)

---

## 1. 問題設定

新しい flow を始めるたびに、ユーザーは手作業で:

1. UUID v7 を生成する(`uuidgen` には v7 が無く、別ツールが必要)
2. `<root>/<uuid>/` ディレクトリを作る
3. `flow.toml` の `uuid` / `created_at` を手で埋める(UUID とディレクトリ名の一致が必須 — `JobFlow` の不変条件)
4. `plan.toml` を flow.toml の JobId に合わせて作る

この手順は退屈でミスを生みやすい(特に「ディレクトリ名 = flow.toml の uuid」の不一致は `FlowRun::read` まで気付けない)。`jm new` でこの定型作業をワンショット化する。

## 2. ゴール / 非ゴール

### Goals

1. `jm new` サブコマンドを追加。新しい UUID v7 を発行し、`<root>/<uuid>/` 配下に **`flow.toml` と `plan.toml`** の編集可能なテンプレートを生成する。
2. 生成 UUID がルート内で衝突しないことを確認する(防御コード — `now_v7()` の実衝突は実質ゼロ)。
3. テンプレートは `examples/simple` 同型の **2-job DAG**(`step1 → step2`, `afterok`)。生成直後に `jm render` / `jm submit --dry-run` がエラーなく通る形にする。
4. `--tag k=v`(繰り返し可)で `[tags]` を埋められる。
5. 既存 CLI の `resolve_root()` / `atomic_write()` を再利用し、書き込みは中途半端な状態を残さない。

### Non-goals

- **`common.toml` の生成・変更には一切関与しない**(v1)。理由: ルートの `common.toml` はクラスタ固有値(partition / project_root)を持ち、既存ファイルの上書きや `REPLACE_ME` 入り stub の自動生成は事故源。`common.toml` 対応は将来 `jm new --with-common` 等で別途切る(本 spec の対象外)。
- experiment DSL / sweep 展開(callers が Python で `JobEdge` を組む — CLAUDE.md "Out of scope" 準拠)。
- 対話的ウィザード / TUI。
- 既存 flow の再生成・migration。
- `jm new` 後の自動 `render` / `submit`(別コマンドの責務)。

## 3. CLI 形

```
jm --root <ROOT> new [--tag <KEY=VALUE>]... [--print-path]
```

| 引数 | 説明 |
|---|---|
| `--root` / `JM_ROOT` | 既存の global 引数。`resolve_root()` で canonicalize(既存ロジック流用)。 |
| `--tag <KEY=VALUE>` | 任意回。`flow.toml` の `[tags]` に反映。`=` が無ければエラー。 |
| `--print-path` | 付与時は stdout に `<root>/<uuid>` の絶対パスのみ出力(`cd "$(jm --root /work new --print-path)"` 用)。非付与時は人間向けに生成内容を表示。 |

`Cmd` enum に `New { tags: Vec<String>, print_path: bool }` を追加し、`main()` の match に分岐を 1 本足す。

## 4. 動作シーケンス

`cmd_new(root: &Path, tags: &[String], print_path: bool) -> anyhow::Result<()>`:

1. `--tag` をパース。各要素を最初の `=` で `(key, value)` に分割。`=` 無し → `anyhow::bail!("invalid --tag: expected key=value, got {raw}")`。重複キーは後勝ち(`BTreeMap` に挿入)。
2. `let uuid = Uuid::now_v7();`
3. `let resolver = PathResolver::new(root);` `let flow_dir = resolver.flow_dir(&uuid);`
4. 衝突確認: `if flow_dir.exists() { anyhow::bail!("flow dir already exists: {}", flow_dir.display()); }` — リトライしない(衝突は実質起きない、起きたら異常事態として落とす)。
5. `tokio::fs::create_dir_all(&flow_dir).await?`
6. テンプレート組み立て(純粋関数):
   - `build_flow_template(&uuid, created_at, &tags) -> String`
   - `build_plan_template() -> String`
   - `created_at` は `chrono::Utc::now()` を RFC3339(`to_rfc3339_opts(SecondsFormat::Secs, true)`)で文字列化。
7. `atomic_write(&resolver.flow_toml(&uuid), flow_str.as_bytes())?` → 失敗時は **作成済み `flow_dir` を `tokio::fs::remove_dir_all` で巻き戻し**てから `?` 伝播(中途半端なディレクトリを残さない)。
8. `atomic_write(&resolver.plan_toml(&uuid), plan_str.as_bytes())?` → 同上の巻き戻し。
9. 出力:
   - `print_path` 時: `println!("{}", flow_dir.display());`
   - 非 `print_path` 時:
     ```
     created flow <uuid>
       <root>/<uuid>/flow.toml
       <root>/<uuid>/plan.toml
     next: edit flow.toml, then `jm --root <root> render <uuid>`
     ```

`atomic_write` は `persistence::atomic_write`(PID サフィックス付き tmp + rename)。`pub(crate)` のため `src/bin/jm.rs` から使えるか確認 → 使えなければ `cmd_render` 等と同様に薄いラッパを通すか、`std::fs::write` ベースの最小アトミック書き込みを CLI 側ローカルに用意する(実装フェーズで判断、設計上はアトミック性が要件)。

## 5. テンプレート内容

### flow.toml

```toml
# Generated by `jm new` on <RFC3339>.
# Schema: gaussian_job_shared::entities::workflow::JobFlow (deny_unknown_fields)
#   uuid          UUID v7 — MUST equal the parent directory name
#   created_at    RFC3339 UTC
#   jobs.<JobId>  JobSpec (program/body/config) + parents[]

uuid       = "<uuid>"
created_at = "<rfc3339>"

[tags]
# free-form key=value tags; populate via `jm new --tag k=v`
<生成時 --tag があれば key = "value" 行を列挙、無ければコメントのみ>

# --- step 1: replace `program` / `body` with the real workload ---
[jobs.step1]
program = "echo"
body    = "echo \"[step1] flow=$JM_FLOW_UUID job=$JM_JOB_ID\"\n"

[jobs.step1.config]
# partition は common.toml の [slurm_default] から自動補完される。
# 個別に上書きする場合のみ partition = "..." を指定。

# --- step 2: runs only if step1 exits 0 ---
[jobs.step2]
program = "echo"
body    = "echo \"[step2] flow=$JM_FLOW_UUID job=$JM_JOB_ID\"\n"

[[jobs.step2.parents]]
from = "step1"
kind = "afterok"

[jobs.step2.config]
```

### plan.toml

```toml
# Generated by `jm new`. Per-JobId params surface in batch.bash as
# `JM_PARAM_<UPPER_NAME>`.
# Schema: job_manager::plan::ExperimentPlan (deny_unknown_fields)

[jobs.step1]
note = "TODO: replace with real render params"

[jobs.step2]
note = "TODO: replace with real render params"
```

不変条件: `flow.toml` の JobId 集合(`step1`, `step2`)と `plan.toml` の `[jobs.*]` キー集合は一致させる。テンプレートはハードコードなので構造的に保証されるが、ユニットテストでも assert する。

## 6. エラーハンドリング

| 状況 | 挙動 |
|---|---|
| root 不在 / 非正規 | `resolve_root()` の canonicalize 失敗メッセージ(既存) |
| `flow_dir` 既存(UUID 衝突) | `anyhow::bail!("flow dir already exists: {path}")`。リトライしない。 |
| `--tag` に `=` 無し | `anyhow::bail!("invalid --tag: expected key=value, got {raw}")` |
| `flow.toml` / `plan.toml` 書き込み失敗 | 作成済み `flow_dir` を `remove_dir_all` で巻き戻してから `?` 伝播 |
| `create_dir_all` 失敗 | `?` 伝播(巻き戻し不要 — 何も作っていない) |

## 7. 配置とテスト

### 配置

- `src/bin/jm.rs` に `Cmd::New` バリアント + `cmd_new()` + ヘルパー `build_flow_template`, `build_plan_template` を追加。
- ヘルパーは I/O を持たない純粋関数(`&Uuid`, `&str`, `&BTreeMap<String,String>` → `String`)。

### テスト

1. **ユニット**(`src/bin/jm.rs` の `#[cfg(test)] mod tests`):
   - `build_flow_template` の出力が `toml::from_str::<JobFlow>` でパースできる(common 注入が要るので、`synth_empty_common` を使い `read_flow` 相当の経路 — もしくは partition 注入後の effective 形でパース確認)。
   - `build_plan_template` の出力が `toml::from_str::<ExperimentPlan>` でパースできる。
   - flow と plan の JobId 集合一致を assert。
   - `--tag` パース: `a=b` → `("a","b")`、`a=b=c` → `("a","b=c")`、`abc`(=無し)→ Err。
2. **統合**(`tests/integration_new.rs`, `assert_cmd`):
   - tempdir を root に `jm new` 実行 → exit 0、stdout に uuid。
   - `<root>/<uuid>/flow.toml` と `plan.toml` が存在。
   - 生成 flow を `FlowRun::read`(`synth_empty_common` 経由)で再ロードでき、jobs が 2 個。
   - `jm new` → 続けて `jm render <uuid>` が exit 0(`examples/simple` 同型なので通るはず)= ラウンドトリップ検証。
   - `--print-path` 時、stdout が `<root>/<uuid>` 1 行のみ。
   - `--tag env=prod` で生成 flow の `tags["env"] == "prod"`。

`MockExecutor` / `InMemoryQuerier` を使い、live SLURM は不要(CLAUDE.md 準拠)。

## 8. トレードオフと採用判断

| 案 | 概要 | 判定 |
|---|---|---|
| **A: deferred-common(採用)** | `common.toml` 非関与。flow + plan のみ。 | 採用。ユーザー指示通り。事故ゼロ。 |
| B: idempotent-common | root に `common.toml` 不在時のみ `REPLACE_ME` 入り stub 生成。 | 却下。生成物をそのまま submit すると `invalid partition: REPLACE_ME` で落ち、「使える状態を作る」期待とズレる。 |
| C: `--with-common` flag | 明示時のみ B の挙動。 | v1 では不要(YAGNI)。将来別 spec で検討可。 |

テンプレートを 1-job にする案も検討したが、job-manager のコア機能は DAG(`parents` / `afterok`)であり、2-job のほうが「何を編集すればよいか」が自明になるため `examples/simple` 同型を採用。

## 9. CLAUDE.md 準拠チェック

- `flow.toml` / `plan.toml` は user input。`jm new` はそれらの**初期生成のみ**行い、以後 `.jm/` 配下しか触らない既存規約と矛盾しない(初回作成は user input の bootstrap であり、上書き・再生成はしない)。
- `jm` は `--no-default-features` でも動く必要 → `cmd_new` は pyo3 に依存しない(`uuid` / `chrono` / `toml` / `tokio::fs` のみ)。
- アトミック書き込み(PID サフィックス tmp + rename)を踏襲。
- Conventional Commits / per-task commit。
- Out of scope(DSL / sweep / per-flow common / TUI)に抵触しない。
